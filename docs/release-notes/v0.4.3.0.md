# 0.4.3.0: pre-configured cloud providers

Feature release: five cloud providers — Moonshot AI (Kimi), Z.ai (GLM), OpenAI, Anthropic, and OpenRouter — are now first-class, pre-configured routing targets with enable/disable toggles, a configurable local/cloud fallback direction, and full UI coverage (CLI, admin API, web dashboard, macOS app). Everything is additive: an existing `config.toml` without a `[cloud]` section behaves exactly as before.

## New

- **`netllm cloud` CLI** — `list`, `enable`/`disable <id>` (`--region`, `--auth`), `set-key <id>` (hidden prompt or `--env VAR`), `fallback <cloud|local|none|on|off>`, `test <id>` (reachability + model catalog probe), and `connect openrouter` (OAuth PKCE sign-in — the only provider here with a sanctioned third-party OAuth flow).
- **`[cloud]` config section** — `cloud.enabled` (master switch), `cloud.fallback` (`"cloud"`: local first then cloud, default and unchanged from prior behavior; `"local"`: cloud first then local; `"none"`: no automatic fallback, only explicit `allow_cloud` policies or backend pins reach cloud), and `[cloud.providers.<id>]` entries (enabled, region/profile, api_format, auth mode, key). See [config.example.toml](../../config.example.toml).
- **Cloud routing engine** — enabled providers materialize into keyed, prunable pool rows; a new `pool.select_backend(prefer_cloud=…)` generalizes the existing `prefer_provider` narrow-then-fallback pattern so every strategy respects `cloud.fallback = "local"` uniformly, without touching per-strategy selection logic.
- **Web dashboard Cloud tab** — master toggle, fallback picker, and a card per provider (enable, region, api_format, write-only key field) under Config → Cloud, backed by the admin config API's new `cloud` slice (keys are never echoed back).
- **macOS Settings → Cloud** — same controls, keys stored in Keychain and injected into the agent process only (never written to `config.toml`); menubar popover shows live cloud status (enabled providers, fallback mode).
- **Provider registry** (`netllm_core.cloud_providers`) — official base URLs, auth modes, and model catalogs for all five providers, sourced from each vendor's docs as of 2026-07-22. Notable caveats surfaced in the CLI/UI: Moonshot's `kimi-k2-*`/`kimi-latest`/`moonshot-v1-*` are discontinued — the registry defaults to `kimi-k3`; Z.ai's GLM Coding Plan keys are contractually restricted to an approved-tools list, so the pay-as-you-go `api` region is the safe default for a router; Anthropic's `plan_token` auth mode (from `claude setup-token`) is opt-in only and documented by Anthropic for Claude Code CI, not general third-party clients.
- **`netllm cloud doctor` checks** — an enabled-but-keyless provider now shows up as a doctor issue with the exact env var or CLI command to fix it.

## Design notes

- Materializing a provider preserves its accumulated health/model catalog across requests instead of rebuilding on every call — only a real config change (key, region, format) triggers a rebuild.
- The legacy env-key cloud inject (`ANTHROPIC_API_KEY`/`OPENAI_API_KEY`, no config needed) keeps working unchanged and is now gated by the same `cloud.enabled` master switch.
- Full design and provider citations: [docs/cloud-providers-plan.md](../cloud-providers-plan.md).

## Upgrade

macOS (menubar, recommended on macOS 26+):

```bash
git clone https://github.com/matthewdcage/llm-swarm-router.git
cd llm-swarm-router && git checkout v0.4.3.0
uv sync && uv pip install venvstacks
apps/netllm-mac/Scripts/build.sh release
packaging/scripts/macos-app-install.sh --source apps/netllm-mac/build/Stage/llm-swarm-router.app
```

**CLI only:** `uv sync && ./netllm init && ./netllm serve` — dashboard at http://127.0.0.1:11400/ui/

Try it:

```bash
./netllm cloud list
./netllm cloud enable moonshot
./netllm cloud set-key moonshot --env MOONSHOT_API_KEY
export MOONSHOT_API_KEY=sk-...
./netllm cloud test moonshot
```
