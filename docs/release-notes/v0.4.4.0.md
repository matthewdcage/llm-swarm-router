# 0.4.4.0: fair load balancing, per-machine concurrency caps, drain

Fix + feature release from a live swarm-utilization report: `auto`/`least_load` was silently favoring the local machine on load ties, and there was no way for a machine to declare its own concurrency ceiling or step out of routing gracefully before a restart.

## Fixed

- **`least_load` no longer always favors local on a tie** — `pool.select_backend`'s `least_load` branch used `min(all_candidates, key=in_flight)` over a local-then-remote list; Python's `min()` returns the first element on a tie, so any exact in_flight tie between a local backend and a peer (the common case at small counts — both idle at 0, both at 1) silently and permanently picked local. Under `"auto"` (which resolves to `least_load` with no shard context), this meant peers only got picked when *strictly* less loaded than every local backend, starving them of concurrent work while local climbed to 7-8+ in flight. Fixed: ties now rotate fairly among the tied candidates. No behavior change when there's a single clear minimum.

## New

- **`agent.max_concurrency`** (0 = unlimited, default) — each machine self-declares a ceiling on its own total concurrent requests, broadcast via heartbeat. Every peer's `least_load`/`local_spillover` selection respects it once that machine hits its own cap — no machine can impose a limit on another, only on itself. A tighter per-backend cap also works locally via `[[routing.backends]] max_concurrency` (e.g. capping one provider process without affecting others on the same machine).
- **`netllm drain [on|off]`** (`POST /netllm/v1/admin/drain`) — ahead of a planned restart or shutdown, tell every peer to stop routing new work here. Runtime-only (never persisted, resets automatically on the next process start); requests already in flight finish normally, nothing is cancelled. A draining peer disappears from every other agent's routing candidates on their next heartbeat.
- Both new fields are broadcast in the same heartbeat payload as everything else (`/netllm/v1/status`), and gracefully default to off/unlimited when a peer is running an older netllm version that doesn't send them yet.

## Upgrade

macOS (menubar, recommended on macOS 26+):

```bash
git clone https://github.com/matthewdcage/llm-swarm-router.git
cd llm-swarm-router && git checkout v0.4.4.0
uv sync && uv pip install venvstacks
apps/netllm-mac/Scripts/build.sh release
packaging/scripts/macos-app-install.sh --source apps/netllm-mac/build/Stage/llm-swarm-router.app
```

**CLI only:** `uv sync && ./netllm init && ./netllm serve` — dashboard at http://127.0.0.1:11400/ui/

Try it:

```bash
./netllm status                 # shows draining state + max_concurrency once set
# set agent.max_concurrency = 8 in config.toml, restart, then on another machine:
./netllm drain on                # tell peers to stop routing new work here
./netllm drain off               # rejoin
```
